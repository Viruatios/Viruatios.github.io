# AGENTS.md

## Project overview

This repository is the source for CuLoo's personal site at `https://viruatios.github.io/`. It is a statically generated, content-driven Astro site with a small React island and browser-side TypeScript for navigation, theming, and GSAP-powered SVG mascot animations.

The site is intentionally lightweight: there is no server runtime, database, authentication layer, test runner, linter, or formatter configured. GitHub Actions builds the static `dist/` directory and deploys it to GitHub Pages.

### Technology stack

- Astro 6 in strict TypeScript mode
- React 19 through `@astrojs/react`
- GSAP 3 for SVG interaction and animation
- Markdown backed by Astro Content Collections
- npm with a committed `package-lock.json`
- GitHub Actions and GitHub Pages; CI uses Node.js 22

## Repository map

- `src/pages/`: file-based routes. Dynamic post and tag routes use `getStaticPaths()` and are generated at build time.
- `src/blog/`: Markdown blog entries. This directory is loaded as the `blog` content collection; it is not a route directory.
- `src/content.config.ts`: the authoritative blog frontmatter schema.
- `src/layouts/`: shared full-document layouts and the Markdown post wrapper.
- `src/components/`: Astro components, one hydrated React greeting, and the inline SVG mascot components.
- `src/scripts/`: browser-side TypeScript for menus, navigation, themes, and mascot animation state machines.
- `src/styles/`: global, entry-page, navigation, and theme-control styles.
- `public/`: unprocessed icons and SVG assets copied to the output root.
- `.github/workflows/deploy.yml`: build, type-check, artifact upload, and GitHub Pages deployment.
- `dist/`, `.astro/`, and `node_modules/`: generated or installed locally; never edit or commit them.
- `Astro - Default-README.md`: preserved starter reference, not the project's authoritative documentation.

## Setup and commands

Run commands from the repository root.

```bash
# Reproduce the lockfile exactly (preferred for validation and CI parity)
npm ci

# Start the development server, normally at http://localhost:4321
npm run dev

# Run Astro's content, component, and TypeScript checks
npx astro check

# Generate the production site in dist/
npm run build

# Serve an existing production build locally
npm run preview
```

Use Node.js 22 when matching CI. `npm install` is acceptable when intentionally changing dependencies and updating `package-lock.json`; otherwise prefer `npm ci`.

There is currently no `test`, `lint`, or `format` script. Do not claim those checks ran, and do not introduce a toolchain solely to satisfy a small unrelated change.

## Architecture and routing

- `BaseLayout.astro` provides the common document shell, header, page heading, footer, global styles, and the early `html.js` marker used to prevent animation flashes.
- `EntryLayout.astro` is the focused home-page shell and uses `entry.css`.
- `MarkDownPostLayout.astro` renders post metadata, tags, image, and Markdown content inside `BaseLayout`.
- `/`, `/about/`, `/blog/`, `/randomNumber/`, and `/404.html` come from files in `src/pages/`.
- `/posts/<id>/` is generated by `src/pages/posts/[...slug].astro` from every `blog` collection entry.
- `/tags/` and `/tags/<tag>/` are generated from `post.data.tags`.
- Navigation intentionally maps `/posts/*` back to the Blog tab and `/tags/*` to the Tags tab.

The deployment target is a GitHub Pages user site at the domain root. Existing internal asset and route URLs therefore begin with `/`. Preserve trailing-slash behavior in navigation comparisons and generated links.

## Blog content rules

Add articles under `src/blog/` as UTF-8 Markdown. Each non-underscored `.md` file is collected by the glob loader and must provide exactly the fields required by `src/content.config.ts`:

```yaml
---
title: "Article title"
pubDate: 2026-01-01
description: "Short summary"
author: "Author name"
image:
  url: "/asset.svg"
  alt: "Meaningful alternative text"
tags: ["Astro", "tutorial"]
---
```

- Keep post assets in `public/` when using root-relative URLs.
- Preserve Chinese text as UTF-8; verify terminal encoding before treating garbled console output as file corruption.
- Tag spelling and capitalization create distinct tag routes. Reuse existing forms intentionally.
- Astro uses the filename-derived content entry ID as the post slug. Renaming a Markdown file changes its public URL.
- Run `npx astro check` after frontmatter or collection-schema changes.

## Component and browser-script conventions

- Prefer Astro components for static UI. Add a framework island only when the feature needs client-side state; choose the least eager suitable `client:*` directive.
- Keep reusable browser behavior in `src/scripts/`; a small page-specific script may remain colocated in its `.astro` page.
- Query DOM elements defensively and narrow their types with `instanceof` or typed selectors before use.
- Scripts may execute more than once during Astro navigation. Preserve the existing `data-*` binding guards and global bootstrap guards when changing mascot behavior.
- Clean up long-lived timers or listeners when a page lifecycle requires it. The random-number page, for example, clears its interval on `pagehide`.
- Keep accessibility state synchronized (`aria-expanded`, `aria-current`, `aria-live`) and retain keyboard-safe native controls.
- Theme choice is stored in `localStorage` under `theme`; supported values are `light` and `dark`. Theme transitions must continue to fall back when View Transitions are unavailable or reduced motion is requested.

## GSAP mascot architecture

The mascot SVG structure and its TypeScript selectors are a coupled interface. Before renaming classes or reorganizing SVG groups, search both the component and corresponding script.

- `CuLooMascot.astro` is controlled by `culooMascot.ts`, which implements a state machine, baseline reset, weighted/cooldown animation registry, blink scheduling, click history, and a high-priority rage state.
- `CuLoo404Mascot.astro` is controlled separately by `culoo404Mascot.ts`.
- Animations must start from and restore a known baseline. Kill or complete active timelines before leaving a component in another state.
- Keep repeat-binding protection and the `data-mascot-ready` reveal flow; they prevent duplicate handlers and initial visual flashes.
- Respect `prefers-reduced-motion` for any new page-level or nonessential animation.
- Prefer transforms and opacity for motion. Test rapid repeated clicks, animation completion, resize behavior, and both themes after animation changes.

## Code style

- Follow the style of the file being edited. Existing Astro markup generally uses tabs; TypeScript files commonly use four-space indentation.
- Use double quotes in TypeScript/TSX and semicolons.
- Keep strict TypeScript types. Avoid adding `any`; existing collection-page `any` annotations are legacy, not a pattern to copy.
- Use `const` by default, small named helpers, early returns, and explicit return types for nontrivial TypeScript functions.
- Keep Astro frontmatter imports at the top, derive page data before the closing `---`, and keep component markup readable.
- Prefer shared styles for cross-page behavior and scoped `<style>` blocks for truly page-local rules.
- Preserve public behavior and Chinese copy unless the task explicitly changes it.

No automatic formatter is configured, so avoid broad whitespace-only rewrites.

## Validation checklist

For every code or content change, run the smallest relevant checks and finish with the same gates used by CI:

```bash
npx astro check
npm run build
```

There is no automated browser suite. Manually validate affected behavior with `npm run dev` or `npm run preview`, including as applicable:

- the changed route and its generated links;
- `/blog/`, one `/posts/<id>/` page, `/tags/`, and a tag detail page after content changes;
- mobile and desktop navigation around the 636px breakpoint;
- menu expansion and active underline behavior;
- light/dark theme persistence and reduced-motion fallback;
- mascot idle, click, repeated-click, and completion states;
- browser console errors and keyboard interaction.

If dependencies change, verify that `package.json` and `package-lock.json` change together and use a clean `npm ci` before handoff.

## Build and deployment

Pushes to `main` and manual workflow dispatches trigger `.github/workflows/deploy.yml`. The workflow:

1. checks out the repository;
2. installs Node.js 22 and runs `npm ci`;
3. runs `npx astro check`;
4. runs `npm run build`;
5. uploads `dist/` and deploys it with GitHub Pages.

Do not hand-edit `dist/`. A successful local build is necessary but does not itself deploy the site.

## Change discipline

- Inspect `git status` before editing and preserve unrelated user work in a dirty worktree.
- Keep changes scoped; avoid opportunistic rewrites of tutorial-era code.
- Never commit secrets. `.env` and `.env.production` are ignored, and this static site should not expose private values in client bundles.
- Treat third-party URLs, embedded personal data, and public assets as user-visible content.
- When behavior, commands, dependencies, routes, or content requirements change, update this file and the human-facing `README.md` where appropriate.

## Definition of done

A change is complete when it is scoped to the request, preserves unrelated work, passes `npx astro check` and `npm run build`, has received relevant manual browser verification, and updates documentation or content schemas when its operating assumptions changed.
